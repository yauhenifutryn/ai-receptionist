## Parallel Claude Code sessions

Two patterns. Pick by task shape, not by ambition.

- **Pattern A — Multiple top-level sessions** (separate tmux/window/IDE tabs): genuinely independent feature work on different files. Our Jenya-vs-Rem team split is the canonical case. Each session is its own conversation; standard PR-merge flow keeps them in sync.
- **Pattern B — Subagents inside one session** via Agent tool: parallel research or independent sweeps. Use `superpowers:dispatching-parallel-agents`. Example: scrape 5 Polish dental sites in parallel to bench Layer 1+2 vs Layer 1+2+3.

Anti-patterns:

- Two top-level sessions both editing the same file → guaranteed conflict.
- Subagent that needs context from another subagent → sequence, do not parallelize.
- Parallelizing for speed on a sequential task → coordination overhead with no throughput gain.

## Hard rules for any code change in this repo

- **Never log patient PII in Vercel logs.** Use structured logging that strips PII before flushing.
- **Audio recordings: never stored.** Audio passes through ElevenLabs and is discarded. Audio saving permanently OFF on every agent.
- **Transcripts: only stored if `consent_flag === true`.** Consent flow runs deterministically at the start of every call. Default to false if unclear.
- **EU regions everywhere.** Vercel `fra1`, Supabase Ireland (`eu-west-1`), Twilio EU media region, LLM in EU.
- **ElevenLabs workspace setting** "Use conversation data for model improvement" → OFF on Day 1.
- **Polish phrasing is canonical**, not literal translations. Every patient-facing string must read naturally to a native Polish speaker.
- **Voice runtime is abstracted** behind `VoiceAgentProvider` interface in `packages/contracts/`. ElevenLabs today, swappable to Vapi or Synthflow tomorrow.
- **Agent escalates, does not improvise**, on anything operationally complex (medical questions, billing, treatment plans, emergencies, NFZ paperwork, complaints).
- **Secrets in `.env.local`**, never committed. `ELEVENLABS_API_KEY`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_ANON_KEY`, `TWILIO_*`, `SMSAPI_*`, `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `FIRECRAWL_API_KEY`.
- **Speed > architectural perfection.** 16 days to Demo Day. Build to working, then refactor only what hurts.

## Knowledge architecture (3 layers, RAG retrieval pulls all)

**Layer 1 — Polish dental ontology** (our authored IP): `apps/backend/ontology/{services,triage,scripts,emergency-keywords,consent}.md`. Version-controlled. Uploaded as agent KB documents. One service per H2 section. Polish synonyms front-loaded. Prices as `Cena: X PLN` lines not tables. Date-stamped. 200-500 token sections.

**Layer 2 — Per-clinic curated KB** (Firecrawl-scraped, consolidated): `data/clinics/<clinic_id>/knowledge.md`. Firecrawl-map + crawl → Claude consolidation → structured markdown matching ontology shape (Klinika, Lekarze, Usługi i ceny, FAQ) → upload to ElevenLabs.

**Layer 3 — Live website connector** (ElevenLabs native, OPTIONAL complement): adds freshness if clinic updates their site post-onboarding. Verified Day 8-9 vs Layer 1+2-only on 20 test queries; ship only if it beats baseline on >70% of categories.

**Hard rule**: scraper consolidation prompt must say _"do not invent prices; mark `unknown` if not in source."_ Validate before upload.

Full architecture detail and indexing rules: `docs/plans/2026-05-14-end-to-end-plan.md` Part 2.

## What we are NOT building

Not a reminder bot. Not "press 1 to confirm." Both inbound and outbound (v2) are conversational. No email collection during calls. No patient-attendance tracking; scope ends at "appointment booked."
